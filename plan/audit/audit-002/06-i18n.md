## Resumen ejecutivo

La app tiene un sistema i18n bien estructurado con ES/EN y hook `useI18n()`. Las traducciones
`es.ts` y `en.ts` tienen el mismo número de secciones de top-level y parecen sincronizadas
(es.ts: 431 líneas, en.ts: 427 líneas — diferencia de 4 líneas por variaciones de longitud de
strings). Sin embargo, 38 strings visibles al usuario están hardcodeados directamente en JSX,
bypasando el sistema i18n. Esto hace la app no traducible en esas secciones.

---

## Hallazgos

### [I18N-001] 38 strings hardcodeados en JSX — bypasean el sistema i18n

> **Fuente:** 🤖 Auto-descubierto (BRF-004)

**Archivo(s):** 8 screens con strings hardcodeados  
**Severidad:** 🟡 Media  
**Impacto:** La app soporta ES/EN pero estos strings no se traducen. Algunos están en inglés
(DEFEATED, KO), otros en español (CERRAR, INICIAR NUEVA TEMPORADA, HERRERÍA, MERCADO,
DESCENDIENDO). Un usuario EN verá estos strings en español y vice versa.

**Muestra de strings hardcodeados:**

**Solución paso a paso:**

1. Agregar las claves faltantes al archivo ES (y EN):
```ts
// src/i18n/translations/es.ts — agregar en la sección correspondiente
battle: {
  // ...existentes...
  defeated: 'DERROTADO',
  ko: 'KO',
  attackAction: 'ATACAR',
},
village: {
  // ...existentes...
  closeBtn: 'CERRAR',
  startNewSeason: 'INICIAR NUEVA TEMPORADA',
  viewWorldLog: 'VER HISTORIAL DE LA TORRE',
},
// etc.
```

2. Reemplazar los literals en los componentes:
```tsx
// ANTES
<Text style={S.defeatedText}>DEFEATED</Text>

// DESPUÉS
const { t } = useI18n();
<Text style={S.defeatedText}>{t('battle.defeated').toUpperCase()}</Text>
```

3. Para abreviaturas fijas como "RNK", "ALI", "MIS", "LOG" en GuildScreen — estas son
   identificadores de UI cortos cuyo significado es el mismo en todos los idiomas.
   Si se decide i18n-izar, agregar como:
```ts
guild: {
  tabRanking: 'RNK',    // es
  tabAlliances: 'ALI',
  tabMissions: 'MIS',
  tabLog: 'LOG',
},
```

**Tiempo estimado:** 2-3 horas  
**Prioridad:** P2 — Necesario para publicación multi-idioma real

### [I18N-002] `es.ts` tiene 4 líneas más que `en.ts` — posible key faltante en EN

**Archivo(s):** `src/i18n/translations/es.ts`, `src/i18n/translations/en.ts`  
**Severidad:** 🟢 Baja  
**Impacto:** La diferencia de 4 líneas puede indicar que hay keys en ES que no están en EN
(o que los strings en EN son simplemente más cortos). El TypeScript `as const` no valida
la sincronía entre archivos — si una key existe en ES pero no en EN, la app no crashea pero
muestra `undefined` al cambiar a inglés.

**Verificación:**
```bash
# Contar keys de top-level que difieren
diff <(grep "^  [a-zA-Z_]" src/i18n/translations/es.ts | sort) \
     <(grep "^  [a-zA-Z_]" src/i18n/translations/en.ts | sort)
```

**Solución:**
1. Crear un script de validación:
```ts
// scripts/validate-i18n.ts
import { es } from '../src/i18n/translations/es';
import { en } from '../src/i18n/translations/en';

function findMissingKeys(obj1: object, obj2: object, path = ''): string[] {
  const missing: string[] = [];
  for (const key of Object.keys(obj1)) {
    const fullPath = path ? `${path}.${key}` : key;
    if (!(key in obj2)) {
      missing.push(fullPath);
    } else if (typeof (obj1 as any)[key] === 'object') {
      missing.push(...findMissingKeys((obj1 as any)[key], (obj2 as any)[key], fullPath));
    }
  }
  return missing;
}

const missingInEN = findMissingKeys(es, en);
if (missingInEN.length > 0) {
  console.error('Keys in ES missing from EN:', missingInEN);
  process.exit(1);
}
console.log('✅ i18n keys synced');
```

2. Ejecutar: `npx ts-node scripts/validate-i18n.ts`

**Tiempo estimado:** 30 min  
**Prioridad:** P3 — Prevención
